Autonomous Mobile Robot · target architecture

The AMR codebase, end to end

A differential-drive robot built as three cooperating worlds: the Raspberry Pi thinks (ROS2 / Python), the Pico reacts (MicroPython, real-time I/O), and the browser console watches. This walks every package, every file, every function — and how they talk once the build is complete.

01

System overview

One robot, split across two computers plus a dashboard. The split isn't arbitrary: Linux is not a real-time OS, so anything that must hit a deadline every few milliseconds — motor PWM, encoder polling, the lift PID, fail-safes — lives on the Pico, which has no OS to pause it. Everything that needs to "think" lives on the Pi.

Pico firmware — the real-time half

Plain MicroPython flashed onto the RP2040 — not a ROS2 package. Two cores run in parallel: core 0 handles serial comms, motor drive and encoder reads; core 1 runs the lift PID independently so the lift can hold arbitrary positions without stalling the comms loop.

How it all interacts

The completed system is one closed loop: commands flow down from the operator to the motors, and sensing flows up from the encoders back to the operator. Nodes never call each other directly — they only agree on a topic name and a message type. Here's the whole loop.

Command path (down) Sensing / telemetry (up)

Lift, battery and fault subsystems (amr_lift, amr_charging, amr_error) attach into this loop at amr_pico_bridge and amr_command.

Services & actions

Services (request → one response) handle one-shot commands: TriggerEstop, GetRobotState, GetPicoStatus, SetLiftTarget, SetDriveLimits, ResetOdometry, AcknowledgeFault.

Actions (goal + feedback + result) handle long-running jobs: NavigateToGoal (drive to a pose, streaming distance-remaining) and MoveLift (reach a height, streaming position). Both are owned by amr_engine as the supervisor.

Where JSON, where typed messages

JSON is used only at the two edges — where ROS2 can't reach: the browser (ws://…:8765) and the OS-less Pico (USB-serial). Everything inside the ROS2 graph is strongly-typed amr_msgs, checked at build time.

amr_command bridges the browser JSON edge; amr_pico_bridge bridges the Pico JSON edge. Between them, no dicts — only compiled message classes.

Takeaway

JSON at the edges, typed messages in the core. The browser and the Pico each speak JSON because neither lives in ROS2 — but the moment data crosses into the graph it becomes a compiled amr_msgs type, so a wrong field is a build error, not a 3 a.m. silent-failure.

Wire protocol — RPi ⇄ Pico

One shared USB-serial port, newline-delimited JSON, UTF-8, 115200 baud. One JSON object per line. Every frame carries type + seq; telemetry frames add ts (the Pico's ticks_ms()). The contract lives in two mirrored files: protocol_codec.py (Pi) and comms_protocol.py (Pico).

Appendix

Bringup order

amr_engine.launch.py is the one command that starts the robot — it launches 8 nodes. SLAM/Nav2 launch separately, swapped between mapping and localization modes.

1amr_pico_bridge — opens the serial port first; nothing moves without it.
2drive_controller + odometry — the motion core (cmd_vel ↔ wheels ↔ odom).
3amr_lift · amr_error · amr_charging — the subsystem nodes.
4amr_command — opens the WebSocket so the console can connect.
5amr_engine — the supervisor state machine, aggregating everyone's diagnostics.
+slam.launch.py / nav2.launch.py — SLAM Toolbox & Nav2, launched on their own.
